# /dev-discipline:systematic-debug — 根本原因にたどり着く体系的デバッグ

バグ・テスト失敗・CI 失敗を、当て推量ではなく**根本原因の特定**を通じて解消するための手順です。
`$ARGUMENTS` に問題の説明があればそれを対象に、無ければ直近の文脈・差分・エラーログから対象を推定します。

## 鉄則(Iron Law)

> **理解していない問題を修正してはならない。**

原因を一文で説明できないうちはコードを変更しない。説明できてから、最小の修正を当てる。

## レッドフラグ(これをやり始めたら立ち止まる)

- 原因が分からないまま「とりあえず」コードを変える / 設定を足す。
- エラーメッセージを読まずに直しにかかる。
- 複数の変更を一度に入れて、何が効いたか分からなくなる。
- `try/catch` やリトライで症状を隠す(原因は残ったまま)。
- 「たぶんここ」で再現確認をせずに修正を確定する。

## 4フェーズ

### 1. 理解(Reproduce & Observe)
- エラーメッセージ・スタックトレース・失敗ログを**最後まで**読む。最初の1行ではなく、根本のエラーを探す。
- 確実な再現手順を確立する(再現できない不具合は直したか判定できない)。
- 「期待する挙動」と「実際の挙動」を1行ずつで書き出す。
- 最近の変更を疑う: `git log --oneline -20`、`git diff`、`git status` で何が変わったか確認する。

### 2. 仮説(Hypothesize)
- 観測事実から、原因の仮説を**1〜3個**立てる。各仮説は「もし X が原因なら、Y が観測されるはず」という検証可能な形にする。
- 切り分けの軸を決める: いつから / どの環境で / どの入力で 起きるか。差分(変わった点)に注目する。

### 3. 検証(Test the hypothesis)
- 仮説を**一つずつ**検証する。ログ追加・最小再現・二分探索(`git bisect` 的な切り分け)・該当コードの精読。
- 一度に1変数だけ動かす。複数同時に変えない。
- 仮説が外れたら捨てて次へ。当たったら「なぜそうなるか」を説明できるまで掘る。

### 4. 修正(Fix at the root & confirm)
- 根本原因に対する**最小の修正**を当てる。症状ではなく原因を直す。
- 再現手順で直ったことを確認する。テストがあれば通す。可能なら原因を閉じ込める回帰テストを追加する(`/dev-discipline:tdd-cycle` を参照)。
- 副作用がないか、関連箇所を確認する。

## 出力フォーマット

```markdown
# systematic-debug

**対象**: [問題の1行要約]
**期待 / 実際**: [期待] / [実際]
**再現手順**: [手順]

**観測事実**: [ログ・差分から分かったこと]
**仮説と検証**:
- 仮説1: … → 検証: … → 結論(採否)
- 仮説2: …

**根本原因**: [一文で]
**修正方針**: [最小の修正と、確認方法]
```

## このリポジトリでよくある例

- **CI が落ちるが手元では通る** → まず差分を疑う。依存更新後に `pnpm-lock.yaml` が更新されていない / `mise` のツールバージョンがズレている、が頻出の根本原因。`git diff` と CI ログのインストール段を突き合わせる。

## 使用例

```bash
/dev-discipline:systematic-debug pnpm install が CI でだけ失敗する
/dev-discipline:systematic-debug
```
