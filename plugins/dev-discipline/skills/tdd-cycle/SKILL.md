# /dev-discipline:tdd-cycle — RED → GREEN → REFACTOR を回す

`$ARGUMENTS` の機能を、テスト駆動で実装します。順序を崩さず、各フェーズの完了条件を満たしてから次へ進みます。

## 鉄則(Iron Law)

> **テストを書く前に実装コードを書いてはならない。**

期待する振る舞いを先に失敗するテストとして書く。テストが「赤」になることを必ず目視してから実装に入る。

## レッドフラグ(これをやり始めたら立ち止まる)

- 実装を先に書いて、後からテストを付け足す(それは TDD ではない)。
- テストを書いたが**失敗することを確認せず**に実装へ進む(間違って常に緑のテストになりがち)。
- 1サイクルで複数の振る舞いを盛り込み、どのテストが何を保証するか曖昧になる。
- 通すために、テストを甘くする/スキップする/期待値をコードに合わせて書き換える。

## サイクル

### 🔴 RED — 失敗するテストを書く
1. 実装したい振る舞いを**1つ**選ぶ(小さく刻む)。
2. その振る舞いを検証するテストを書く。既存のテスト構成・命名・フレームワーク(RSpec / Vitest / Jest 等)に合わせる。
3. テストを実行し、**意図した理由で失敗する**ことを確認する(未実装による失敗であって、書き間違いによる失敗ではないこと)。

### 🟢 GREEN — 最小実装で通す
4. そのテストを通すための**最小限**の実装を書く。先回りして余計な汎用化をしない。
5. テストを実行し、対象テストと既存テストが**すべて緑**になることを確認する。

### 🔧 REFACTOR — 整える
6. 緑を保ったまま、重複の除去・命名改善・構造整理を行う。振る舞いは変えない。
7. 各リファクタの後にテストを再実行し、緑が維持されていることを確認する。

→ 次の振る舞いがあれば 🔴 に戻る。

## 進め方の指針

- **1サイクル=1振る舞い**。大きな機能は小さなサイクルに分割する。
- テストファースト・最小実装・緑維持の3点を毎サイクル守る。
- フレームワークはプロジェクトの既存設定を尊重する(新しいテストランナーを持ち込まない)。

## 出力フォーマット

```markdown
# tdd-cycle: [機能名]

## サイクル 1: [振る舞い]
- 🔴 RED: [追加したテスト] → 失敗を確認
- 🟢 GREEN: [最小実装] → 全テスト緑
- 🔧 REFACTOR: [整理した点 / 無ければ「なし」] → 緑維持

## サイクル 2: …
```

## 使用例

```bash
/dev-discipline:tdd-cycle メールアドレスの形式チェック関数を追加
/dev-discipline:tdd-cycle 在庫が0のとき購入を拒否する
```
