# /effort-router — タスク規模に応じた effort/モデル選定

着手しようとしているタスクの規模・不確実性・リスクを見積もり、**過不足のない effort レベル・モデル・実行モード**を推奨します。
目的は「小さいタスクに大きな計算を使わない／大きいタスクに小さな計算で挑まない」ことで、品質を保ちながらトークンを節約することです。

`$ARGUMENTS` にタスクの説明があればそれを対象に、無ければ直近のユーザー指示や現在の差分(`git status` / `git diff`)から対象タスクを推定します。

## 前提: 調整できるつまみ (Claude Code)

| つまみ | コマンド | 効果 |
|---|---|---|
| effort レベル | `/effort <level>` | `low` / `medium` / `high`(既定) / `xhigh` / `max`。思考の深さ＝トークン消費を制御 |
| ultracode | `/effort` → ultracode | `xhigh` 相当 ＋ **dynamic workflows** の自動オーケストレーション (セッション限定) |
| モデル | `/model <name>` | `haiku`(軽量) / `sonnet`(中) / `opus`(高) |
| fast mode | `/fast` | Opus を同品質で最大2.5倍高速化。**1トークン単価は上がる**。対話的反復向き、長時間自律処理には不向き |
| プランモード | Plan mode | 実装前に計画を立て、承認を得てから動く |
| 一発だけ熟考 | `ultrathink` をプロンプトに含める | セッションの effort を変えずに、その1ターンだけ深く推論させる |

> 注: モデルを切り替えると effort はそのモデルの既定値に戻る。`low` などを使いたい場合は `/model` の**後**に `/effort` を設定する。
> `max` と `ultracode` はセッション限定で永続しない。

## 判定軸

タスクを次の4軸で評価します。

1. **規模 (scope)**: 触るファイル/システムの数。1行か、単一ファイルか、横断的か。
2. **不確実性 (uncertainty)**: 方針が明確か、設計判断や調査が要る探索的タスクか。
3. **可逆性/リスク (reversibility)**: 失敗の戻しやすさ。本番・データ・外部影響があるか。
4. **反復性 (iteration)**: 短いフィードバックループを高速に回したいか(対話的デバッグ等)。

## ルーブリック

| ティア | 特徴 | effort | モデル | fast | プランモード | dynamic workflow | 例 |
|---|---|---|---|---|---|---|---|
| **T0 雑務** | 1行・typo・単純質問・完全に可逆 | `low` | haiku / sonnet | 対話時◯ | 不要 | 不要 | typo修正、事実1問、設定値の確認 |
| **T1 小** | 単一ファイル・スコープ明確・低リスク | `medium` | sonnet | 反復時◯ | 不要 | 不要 | 小バグ修正、小機能、lint対応 |
| **T2 中** | 複数ファイル・方針はほぼ固まっている | `high`(既定) | sonnet / opus | 反復時◯ | 推奨 | 任意 | リファクタ、機能実装、renovate/mise設定、CI修正 |
| **T3 大** | 横断的・設計判断あり・不確実・高リスク | `xhigh` / `max` | opus | ✕(長時間自律) | 必須 | 検討 | DB/パッケージマネージャ移行、アーキ変更、CI基盤刷新 |
| **T4 探索/監査** | 広域調査・網羅監査・多観点レビュー | **ultracode** | opus | ✕ | 必須級 | **必須** | 大規模監査、横断調査、多観点コードレビュー |

### 追加の判断

- **1ステップだけ難しい**: 全体は T1〜T2 だが特定の設計判断だけ難しい → effort は上げず、その箇所で `ultrathink` を使う。
- **コンテキストが重い調査**: 多数のファイルを読む必要がある → メインの文脈を汚さないよう**サブエージェント/Workflow に調査を外出し**し、結論だけ受け取る(トークン効率↑)。
- **コスト最優先のバッチ/自律処理**: fast mode は使わず(単価高)、effort も必要最小限に。
- **対話的な高速反復**: fast mode ◯。ただしセッション途中の有効化は累積コンテキストに一度課金されるため、最初から有効が効率的。

## 実行手順

1. **対象タスクの特定**: `$ARGUMENTS` または直近の指示・差分から、これから行うタスクを1〜2文で要約する。
2. **4軸の評価**: 規模・不確実性・可逆性・反復性を一言ずつ評価する。曖昧なら不明点を述べる。
3. **ティア判定**: ルーブリックから T0〜T4 を選ぶ。境界上なら、可逆性が低い/不確実性が高いほど上のティアへ寄せる。
4. **推奨の提示**: 下記フォーマットで、推奨設定と**適用コマンド**、理由を出力する。
5. **AI 自身の振る舞いへの反映**: 推奨が出たら、AI 側で制御できる範囲(プランモードに入る、Workflow を使う、`ultrathink` する、調査をサブエージェントに外出しする)は自律的に適用してよい。`/effort` `/model` `/fast` のようなセッション設定はユーザーに適用を促す。

## 出力フォーマット

```markdown
# effort-router 判定

**対象タスク**: [1〜2文の要約]

**評価**
- 規模: [一言]
- 不確実性: [一言]
- 可逆性/リスク: [一言]
- 反復性: [一言]

**判定: T[0-4] — [ティア名]**

**推奨設定**
| 項目 | 推奨 | 適用 |
|---|---|---|
| effort | [level] | `/effort [level]` |
| モデル | [model] | `/model [model]` |
| fast mode | [on/off] | [`/fast` / 不要] |
| プランモード | [必要/不要] | [Plan mode で開始 / 不要] |
| dynamic workflow | [必要/任意/不要] | [ultracode を有効化 / 不要] |

**理由**: [なぜこのティア・設定か。境界判断があれば明記]
**補足**: [ultrathink を使う箇所、調査の外出し、コスト上の注意など該当すれば]
```

## 使用例

```bash
# タスクを渡して見積もり
/effort-router renovate設定にcooldownとautomergeを追加してPR作成

# 直近の指示/差分から自動推定
/effort-router
```

## 設計思想

- **既定は high で十分**。多くのコーディングタスクは `high`(Opus 4.8 既定)で過不足ない。安易に `max` へ上げない(過剰思考でトークンを浪費しうる)。
- **下げる勇気**。スコープが明確で可逆な作業は `low`/`medium` ＋ 軽量モデルに**下げて**コストを削る。
- **上げるのは大物だけ**。横断・設計・高リスク・広域監査にだけ `xhigh`/`max`/ultracode を使う。
- **局所的な熟考は ultrathink**、**重い調査は外出し**。セッション全体の effort を上げずにピンポイントで深掘りするのがトークン効率の要。
