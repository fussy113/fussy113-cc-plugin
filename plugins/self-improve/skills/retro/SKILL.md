# /retro — セッションの振り返りと教訓の永続化

このセッションで得られた**再利用できる教訓**を抽出し、永続メモリへ書き戻します。
自動記録される Stop フックのジャーナル(「何をしたか」)に対して、`/retro` は
「**何を学んだか / 次回どう振る舞うべきか**」を残す、フィードバックループの書き込み側です。

`$ARGUMENTS` が指定された場合は、その焦点(例: CI、renovate、フロントエンド)に関する教訓を重点的に抽出します。

## 前提: 書き込み先の決定

教訓は以下の優先順位で永続化します。

1. **Claude Code のネイティブメモリ** (環境に永続ファイルメモリが提供されている場合)
   - その `memory/` ディレクトリに1ファイル=1事実で書き込み、`MEMORY.md` に索引を1行追加する。
   - これがあれば次回セッションで自動的に想起される(フィードバックループが閉じる)。
2. **プロジェクトの `CLAUDE.md`** (メモリ機能が無い、またはプロジェクト全体に効かせたいルールの場合)
   - リポジトリ直下の `CLAUDE.md` に追記する。チーム共有・コミット対象になる点に注意。

メモリ機能の有無が分からない場合は、まずメモリへの書き込みを試み、できなければ `CLAUDE.md` にフォールバックしてください。

## 実行手順

### 1. セッションの再構成

このセッションで実際に起きたことを把握します。

```bash
git log --oneline -10
git diff --stat HEAD
git status --short
```

加えて、会話の流れから以下を洗い出してください。
- ユーザーが**訂正・却下**した提案や、やり直しになった箇所
- 何度も**つまずいた / 再発した**問題(同じエラー、同じ手戻り)
- ユーザーが示した**好み・方針**(命名、ツール選択、進め方、レビュー観点など)
- このプロジェクト固有の**前提・落とし穴**(コードを読めば分かることは除く)

### 2. 教訓の抽出と取捨選択

抽出候補について、**残す価値があるか**を厳しく判定します。

**残す (durable, 再利用可能):**
- 次回以降も役立つユーザーの恒常的な好み・方針 → `feedback`
- このプロジェクトの継続中の制約・ゴール・固有の落とし穴 → `project`
- ユーザー自身に関する事実(役割・専門・環境) → `user`
- 外部リソースへのポインタ(URL・ダッシュボード・チケット) → `reference`

**残さない (このセッション限り / 自明):**
- コード構造・git 履歴・既存 CLAUDE.md から分かること
- 一度きりの事象、今回の会話だけで意味を持つ内容
- 当たり障りのない一般論

### 3. 既存メモリとの重複チェック

書き込む前に、同じ事実が既に記録されていないか確認します。
- 既存の `MEMORY.md` / `memory/*.md` を読み、近い内容があれば**新規作成せず更新**する。
- 誤りと判明した古いメモリがあれば削除・修正する。

### 4. メモリへの書き込み

各教訓を1ファイル=1事実で書き込みます。frontmatter は次の形式に従ってください。

```markdown
---
name: <short-kebab-case-slug>
description: <一行要約。想起時の関連判定に使われる>
metadata:
  type: user | feedback | project | reference
---

<事実の本文。feedback / project では続けて **Why:** と **How to apply:** を書く。
関連メモリは [[other-name]] でリンクする。>
```

- `feedback` / `project` は必ず **Why(なぜそうするか)** と **How to apply(次回どう適用するか)** を添える。
- 相対日付は絶対日付に変換する(例: 「先週」→ 実際の日付)。
- 書き込み後、`MEMORY.md` に `- [タイトル](file.md) — 一言フック` を1行追加する。

`CLAUDE.md` にフォールバックする場合は、プロジェクト全体ルールとして簡潔な箇条書きで追記します。

### 5. サマリーの出力

以下の形式で結果を報告してください。

```markdown
# /retro 実行結果

## 書き込んだ教訓
| 種別 | 要約 | 書き込み先 |
|---|---|---|
| feedback | [一行要約] | memory/<slug>.md |
| project | [一行要約] | CLAUDE.md |

## 更新したメモリ
- [既存の何を更新したか]

## 残さなかった候補と理由
- [候補] — [一度きり/自明 などの理由]
```

## 運用のヒント

- **タイミング**: まとまった作業や手戻りがあったセッションの終わりに実行すると効果的。
- **粒度**: 1ファイル1事実を守る。盛り込みすぎたメモリは想起時のノイズになる。
- **保守的に**: 確信が持てない教訓は書かない。少数の確かな教訓のほうが、多数の曖昧なメモリより価値が高い。
- **フィードバックループ**: ここで書いた `feedback` / `project` は次回以降のセッション冒頭で想起され、`self-improve` の Stop ジャーナルと合わせて「記録 → 学習 → 次回適用」のループを閉じる。
