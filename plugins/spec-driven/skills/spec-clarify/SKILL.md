# /spec-driven:spec-clarify — 仕様取り込みから実装方針策定まで

渡された仕様要件を取り込み、**壁打ち対話で曖昧さを潰してから**、実装方針ドキュメントを `docs/spec/` に策定します。
目的は「何を作るか・何を作らないか・どうなれば完了か」をユーザーと合意し、後続の実装(`/spec-driven:spec-implement`)が迷わず進められる土台を作ることです。

`$ARGUMENTS` に仕様(テキスト本文・GitHub Issue の URL/番号・Notion URL・その他 URL)があればそれを取り込み対象にします。無ければユーザーに入力を促します。

## 鉄則(Iron Law)

> **曖昧な仕様のまま実装方針を確定してはならない。**

受け入れ条件とスコープ外がユーザーと言語化・合意できるまで、方針ドキュメントを確定しない。推測で空欄を埋めない —— 分からないことは質問する。

## レッドフラグ(これをやり始めたら立ち止まる)

- 質問せず、自分の推測で要件の空欄を埋めている。
- ユーザーが言っていないスコープを勝手に広げている / 縮めている。
- 受け入れ条件(どうなれば完了か)が定義されないまま方針を書き始めている。
- 取り込んだ仕様の内容を確認せず、URL を開いただけで分かったつもりになっている。
- 非機能要件(性能・セキュリティ・保守性)やエラー時の挙動を一度も話題にしていない。

## 実行手順

### 1. ソースの判定と仕様の取り込み

`$ARGUMENTS` を次の優先順位で判定し、本文を取得する。

- **GitHub Issue**(`#<番号>` または `https://github.com/<owner>/<repo>/issues/<番号>`):
  ```bash
  # URL 指定
  gh issue view https://github.com/<owner>/<repo>/issues/<番号> --comments
  # カレントリポジトリの番号指定
  gh issue view <番号> --comments
  ```
  本文だけでなくコメント欄の議論も取り込む。

- **Notion**(`notion.so` / `notion.site` を含む URL)は3段フォールバック:
  1. Notion MCP(`mcp__claude_ai_Notion__*`)が利用可能ならそれでページ内容を取得する。
  2. 使えなければ `WebFetch` でページ URL を取得する(公開ページならこれで読める)。
  3. 非公開で取得できなければ、ユーザーに「このページは非公開のようです。本文をコピーして貼り付けてください」と依頼する。

- **その他の URL**(`http(s)://`):`WebFetch` で内容を取得する。

- **プレーンテキスト**:そのまま仕様本文として扱う。

- **引数なし**:「仕様要件をテキストまたは URL(GitHub Issue / Notion など)で教えてください」と促す。

取得した仕様は、まず1〜3文に要約してユーザーに提示し、「この理解で合っているか」を確認する。

### 2. 確認項目の洗い出し

取り込んだ仕様を読み、次の8観点で**曖昧な点・未定義の点**を洗い出す。

1. **機能境界**: 何をするか / しないか。
2. **前提・制約**: 言語・フレームワーク・既存依存・互換性の制約。
3. **受け入れ条件(AC)**: 「これが満たされれば完了」と言える条件。
4. **異常系・エッジケース**: 入力の境界値、失敗時の挙動。
5. **非機能要件**: 性能・セキュリティ・保守性などの期待値。
6. **スコープ外**: 今回はやらないことの明示。
7. **既存コードとの統合方針**: 新規追加 / 既存拡張 / 置換 のどれか(`Glob`/`Grep`/`Read` で関連実装を確認する)。
8. **優先度と段階化**: MVP の範囲と、将来拡張に回す部分の切り分け。

### 3. ユーザーとの壁打ち

洗い出した不明点を、観点ごとにユーザーへ質問する。一度に詰め込みすぎず、重要な分岐から確認する。
回答を踏まえて理解を更新し、**収束条件**に達するまで往復する。

> 収束条件: 受け入れ条件とスコープ外がユーザーと合意でき、実装方針(アーキ上の決定とファイル構成案)が一意に書ける状態。

### 4. 実装方針ドキュメントの保存

合意できたら、方針ドキュメントを `docs/spec/` に保存する。

```bash
date +%Y%m%d   # 日付プレフィックスに使う
```

- ファイル名: `docs/spec/<YYYYMMDD>-<slug>.md`(`slug` はタイトルを kebab-case 化したもの)。
- `docs/spec/` ディレクトリが無くても Write が親ごと作成する。
- 下記「出力フォーマット」に従って Write し、**保存先パスをユーザーに提示**する。最後に「実装に進むときは `/spec-driven:spec-implement <このパス>` を実行してください」と案内する。

## 出力フォーマット

`docs/spec/<YYYYMMDD>-<slug>.md` の中身:

```markdown
# 仕様: <タイトル>

## 取り込み元
- ソース種別: [テキスト / GitHub Issue / Notion / URL]
- 参照: [URL または「テキスト入力」]

## 背景・目的
[なぜこれを作るのか。解決したい課題]

## 機能要件
- [やること1]
- [やること2]

## 非機能要件
- [性能 / セキュリティ / 保守性などの期待値。該当なければ「特になし」]

## スコープ外
- [今回はやらないこと]

## 受け入れ条件
- [ ] [満たすべき条件1]
- [ ] [満たすべき条件2]

## 実装方針
### アーキテクチャ上の決定
- [採用する設計・採らなかった選択肢と理由]

### ファイル/モジュール構成(案)
- [新規/変更するファイルと役割]

### 技術的注意点
- [既存コードとの統合、互換性、リスク]

## タスク分解
| # | タスク | 規模感 |
|---|---|---|
| 1 | [タスク] | [小/中/大] |
| 2 | [タスク] | [小/中/大] |
```

## エラーハンドリング

- **GitHub Issue が見つからない / `gh` 未認証**: `gh auth status` を案内し、URL/番号の指定ミスがないか確認する。
- **Notion ページが非公開**: 手順1の3段フォールバックに従い、最終的に本文の貼り付けを依頼する。
- **WebFetch に失敗**: URL の到達性を確認し、取得できない場合は内容の貼り付けを依頼する。

## 注意事項

- スコープ外を勝手に決めない。広げる/縮める判断は必ずユーザーに確認する。
- 受け入れ条件が合意できないうちは方針ドキュメントを確定しない(鉄則)。
- このスキルは**方針策定まで**を担う。コードの編集は行わない —— 実装は `/spec-driven:spec-implement` に渡す。

## 使用例

```bash
/spec-driven:spec-clarify https://github.com/owner/repo/issues/42
/spec-driven:spec-clarify https://www.notion.so/xxxxxxxx
/spec-driven:spec-clarify ユーザー認証機能を追加したい。メール+パスワードで、Google ログインも対応したい
```
